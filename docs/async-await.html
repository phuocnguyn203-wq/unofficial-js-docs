Asynchronous JavaScript

Async/Await

async/await, added in ES2017, is syntax sugar over promises that lets asynchronous code read almost exactly like synchronous code — no .then() chains required. Underneath, it's still promises all the way down.

The async keyword#

Adding async before a function declaration does two things: it lets you use await inside the function, and it makes the function always return a promise, even if you just return a plain value.

JavaScript
async function getGreeting() {
  return "Hello!";
}

getGreeting(); // a Promise, not the string directly
getGreeting().then(msg => console.log(msg)); // "Hello!"

The await keyword#

await pauses execution of the current async function (not the whole program) until the promise to its right settles, then unwraps the result. If the promise rejects, await throws — which is what makes try/catch work naturally with it.

JavaScript
async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);
  const user = await response.json();
  return user;
}

Compare this directly to the promise-chain version of the same logic:

JavaScript
// Promise chain
function getUser(id) {
  return fetch(`/api/users/${id}`)
    .then(response => response.json())
    .then(user => user);
}

// async/await — same behavior, reads top to bottom
async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);
  return await response.json();
}

Note

await can only be used inside a function marked async (modern JavaScript modules also allow "top-level await" directly at the top of a module file). Using it elsewhere is a syntax error.

Error handling: try/catch#

Because a rejected await throws, you handle async errors with the exact same try/catch you'd use for synchronous code (see Error Handling & Debugging) — no special async-only syntax to learn.

JavaScript
async function getUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) {
      throw new Error(`Request failed: ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    console.error("Could not load user:", error.message);
    return null;
  }
}

A common pitfall: accidentally serial awaits#

Every await pauses the function until that specific promise resolves. If you await three independent requests one after another, you needlessly force them to run in sequence instead of in parallel.

JavaScript
// Slow — each request waits for the previous one to finish first
async function loadAll() {
  const user = await fetchUser();       // waits ~1s
  const posts = await fetchPosts();     // then waits another ~1s
  const comments = await fetchComments(); // then another ~1s
  return { user, posts, comments };     // total: ~3s
}

If the three requests don't actually depend on each other, start them all at once and only await the combined result with Promise.all() (see Callbacks & Promises):

JavaScript
// Fast — all three requests fire immediately, running in parallel
async function loadAll() {
  const [user, posts, comments] = await Promise.all([
    fetchUser(),
    fetchPosts(),
    fetchComments(),
  ]);
  return { user, posts, comments }; // total: ~1s, limited by the slowest request
}

Tip

Only await sequentially when a later request genuinely needs data from an earlier one (like fetching posts that require a user ID first). Otherwise, kick off independent requests together and await them as a group.

It's still promises underneath#

async/await doesn't introduce a new concurrency model — it's parsed into the same promise machinery from the previous two chapters. That means everything you learned about the event loop still applies: an await point yields control back to the event loop and resumes as a microtask once its promise settles.

JavaScript
console.log("1: start");

async function delayedLog() {
  await null; // even awaiting a non-promise value yields once, as a microtask
  console.log("3: after await");
}

delayedLog();
console.log("2: sync code after calling delayedLog");

// "1: start"
// "2: sync code after calling delayedLog"
// "3: after await"

Key Takeaways

  • An async function always returns a promise; await pauses that function (not the whole program) until a promise settles.
  • awaiting a rejected promise throws, so ordinary try/catch handles async errors.
  • Sequential awaits on independent operations waste time — start them together and await Promise.all(...) instead.
  • async/await is syntax sugar over promises; the event loop and microtask rules from the previous chapters still apply underneath.
