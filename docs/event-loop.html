Asynchronous JavaScript

The Event Loop

JavaScript runs on a single thread — it can only do one thing at a time. Yet it downloads data, runs timers, and responds to clicks without ever freezing the page. The event loop is the mechanism that makes that possible, and understanding it makes every async pattern in this guide click into place.

The call stack#

The call stack is where JavaScript tracks what function is currently running. Every time a function is called, a frame is pushed onto the stack; when it returns, its frame is popped off. JavaScript can only ever execute the function on top of the stack — this is what "single-threaded" actually means in practice.

JavaScript
function multiply(a, b) {
  return a * b;
}
function square(n) {
  return multiply(n, n);
}
function printSquare(n) {
  console.log(square(n));
}

printSquare(5);
// stack grows: printSquare -> square -> multiply
// then unwinds as each function returns its result

If code keeps calling deeper and deeper without returning (like a function that calls itself forever), the stack fills up and the engine throws a RangeError: Maximum call stack size exceeded — a stack overflow.

So how does JavaScript avoid blocking?#

If the call stack is busy, nothing else can run — including things like a setTimeout callback or the response to a network request. Slow operations (timers, network requests, file access, user events) aren't handled by the JavaScript engine itself at all. They're handed off to APIs provided by the environment — the browser or Node.js — which run them in the background, completely outside the call stack.

JavaScript
console.log("1: start");

setTimeout(() => {
  console.log("2: timeout finished");
}, 1000);

console.log("3: end");

// Logs, in order:
// "1: start"
// "3: end"
// "2: timeout finished"   <- appears ~1 second later

setTimeout doesn't pause anything. It hands the timer off to the browser, and immediately continues to the next line. A second later, once the timer finishes in the background, its callback needs to run — but it can't just interrupt the stack. It waits in line.

The callback queue and the event loop#

When a background task (a timer, a finished network request) completes, its callback is placed in a queue, not run immediately. The event loop is a simple, continuously running check:

  1. Is the call stack completely empty?
  2. If yes, take the first callback waiting in the queue and push it onto the stack to run.
  3. Repeat, forever.

This is why, in the example above, "3: end" always logs before "2: timeout finished" — even with a timeout of 0. The queued callback can only run once the stack is empty, which means after all currently running synchronous code has finished.

JavaScript
console.log("first");
setTimeout(() => console.log("third"), 0);
console.log("second");
// "first", "second", "third" — even a 0ms timeout waits for the stack to clear

Microtasks vs. macrotasks#

There isn't just one queue — there are two, with different priority. Promises (see Callbacks & Promises) schedule their callbacks on the microtask queue, which the event loop always fully drains before touching the regular macrotask queue that setTimeout and DOM events use.

JavaScript
console.log("1: sync");

setTimeout(() => console.log("2: macrotask (setTimeout)"), 0);

Promise.resolve().then(() => console.log("3: microtask (promise)"));

console.log("4: sync");

// Order:
// "1: sync"
// "4: sync"
// "3: microtask (promise)"    <- microtasks run before the next macrotask
// "2: macrotask (setTimeout)"

The rule: after each piece of synchronous code finishes, the event loop empties the entire microtask queue before it processes even one macrotask. This is why promise callbacks reliably run sooner than setTimeout callbacks, even a 0ms one.

Note

async/await (covered two chapters from now) is built entirely on promises, so everything on this page applies to it too — an await point is, under the hood, a microtask.

Putting it all together#

JavaScript
console.log("A");

setTimeout(() => console.log("B"), 0);

Promise.resolve()
  .then(() => console.log("C"))
  .then(() => console.log("D"));

console.log("E");

// A, E   — synchronous code runs first, top to bottom
// C, D   — then the whole microtask queue drains (even chained .then calls)
// B      — only then does the event loop move on to the macrotask queue

Key Takeaways

  • JavaScript runs on one call stack and can only execute one thing at a time.
  • Slow operations (timers, network requests) are handled outside the engine by the browser/Node, which queues their callbacks instead of running them immediately.
  • The event loop moves queued callbacks onto the stack only once the stack is completely empty.
  • Promise callbacks (microtasks) always run before the next timer or event callback (macrotask) — the whole microtask queue drains first.
