Design Patterns

Design patterns are named, reusable solutions to problems that come up again and again. You've actually already met the ideas behind several of these in earlier chapters — this page names them explicitly and shows the classic shape of each.

The module pattern#

You met this exact idea in Closures: use a function's scope to keep some state private, exposing only a curated public interface. Before native ES modules existed, this was wrapped in an IIFE (Immediately Invoked Function Expression) to run once and produce a single object.

JavaScript
const counterModule = (function () {
  let count = 0; // private — inaccessible from outside

  return {
    increment() { return ++count; },
    reset() { count = 0; },
  };
})(); // <- invoked immediately

counterModule.increment(); // 1
counterModule.increment(); // 2
counterModule.count;        // undefined — truly private

Native export/import modules (see Modules) have mostly replaced the IIFE version for organizing files, but the underlying idea — closures for privacy — is exactly the same one used inside class private fields and countless libraries.

The singleton pattern#

Ensures a piece of state or functionality has exactly one shared instance across your whole app, instead of being recreated everywhere it's needed.

JavaScript
const AppConfig = (function () {
  let instance;

  function createInstance() {
    return { theme: "dark", version: "1.0" };
  }

  return {
    getInstance() {
      if (!instance) instance = createInstance();
      return instance;
    },
  };
})();

const config1 = AppConfig.getInstance();
const config2 = AppConfig.getInstance();
config1 === config2; // true — the exact same object, created only once

Note

In practice, a module's top-level state often is a singleton already — since a given module only runs once and every import shares the same exported values, you rarely need to hand-write this pattern explicitly in modern JavaScript.

The factory pattern#

A function that creates and returns objects, hiding the setup details behind a simple call — you've seen this shape already in Closures with createBankAccount.

JavaScript
function createUser(name, role) {
  return {
    name,
    role,
    isAdmin: role === "admin",
    greet() {
      console.log(`Hi, I'm ${name}`);
    },
  };
}

const ava = createUser("Ava", "admin");
const noah = createUser("Noah", "member");

Factories are a lightweight alternative to classes when you don't need inheritance — just plain objects with a consistent shape, built by a single function.

The observer pattern (pub/sub)#

One piece of code ("the subject") maintains a list of interested listeners ("observers") and notifies all of them when something happens. This is exactly how DOM events work (see Events & Delegation) — this pattern is that same idea, generalized to any kind of event, not just browser ones.

JavaScript
function createEmitter() {
  const listeners = {};

  return {
    on(event, callback) {
      (listeners[event] ??= []).push(callback);
    },
    emit(event, data) {
      (listeners[event] || []).forEach(callback => callback(data));
    },
  };
}

const emitter = createEmitter();

emitter.on("userLoggedIn", user => console.log(`Welcome, ${user.name}`));
emitter.on("userLoggedIn", user => console.log(`Logging analytics for ${user.name}`));

emitter.emit("userLoggedIn", { name: "Ava" });
// "Welcome, Ava"
// "Logging analytics for Ava"

This pattern decouples the code that triggers an event from the code that reacts to it — the emitter doesn't need to know or care who's listening, or how many listeners there are.

Key Takeaways

  • The module pattern uses a closure to keep state private while exposing a small public interface — the same idea behind native ES modules.
  • A singleton guarantees exactly one shared instance; native modules often give you this automatically.
  • A factory function builds and returns objects with a consistent shape, without needing a class or inheritance.
  • The observer (pub/sub) pattern decouples an event source from its listeners — the same model DOM events use.
