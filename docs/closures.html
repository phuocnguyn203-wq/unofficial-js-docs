Scope, Closures & this

Closures

A closure is what you get when a function "remembers" the variables from where it was created, even after that outer code has already finished running. It sounds abstract, but you've almost certainly used one already — this page makes the mechanism concrete.

A first example#

JavaScript
function makeCounter() {
  let count = 0;

  return function () {
    count++;
    return count;
  };
}

const counter = makeCounter();
counter(); // 1
counter(); // 2
counter(); // 3

Look closely at what's happening: makeCounter() runs and returns. Normally, count would be thrown away the moment the function finishes — it's a local variable, out of scope. But the inner function that got returned still references count, so JavaScript keeps it alive. Every time you call counter(), it's reading and updating that same, private count variable.

That inner function, bundled together with the variables it needs from its birthplace, is the closure.

Why this happens: it's just the scope chain#

This isn't a special case bolted onto the language — it falls directly out of the scope chain from the previous chapter. A function always keeps a live link to the scope it was defined in (not the scope it's called from). As long as anything — anywhere — still holds a reference to that inner function, the outer variables it depends on can't be garbage collected.

JavaScript
function outer() {
  const secret = "shh";

  function inner() {
    console.log(secret); // inner "closes over" secret
  }

  return inner;
}

const revealed = outer(); // outer() has already finished running
revealed(); // "shh" — the closure kept `secret` alive

Practical use case: private state#

Closures are the main way JavaScript achieves privacy without a dedicated private keyword (before class private fields — see Classes). Anything not returned from the outer function is completely inaccessible from the outside.

JavaScript
function createBankAccount(initialBalance) {
  let balance = initialBalance; // not accessible from outside

  return {
    deposit(amount) {
      balance += amount;
      return balance;
    },
    withdraw(amount) {
      if (amount > balance) throw new Error("Insufficient funds");
      balance -= amount;
      return balance;
    },
    getBalance() {
      return balance;
    },
  };
}

const account = createBankAccount(100);
account.deposit(50);   // 150
account.getBalance();  // 150
account.balance;       // undefined — there's no direct access at all

Practical use case: function factories#

A closure can "pre-fill" some arguments and hand back a more specific function — useful for configuration-style code.

JavaScript
function multiplyBy(factor) {
  return function (n) {
    return n * factor;
  };
}

const double = multiplyBy(2);
const triple = multiplyBy(3);

double(5); // 10
triple(5); // 15

Practical use case: event handlers that remember context#

JavaScript
function setupButton(label) {
  const button = document.createElement("button");
  button.textContent = label;

  button.addEventListener("click", function () {
    // this callback closes over `label`, long after setupButton() returned
    console.log(`${label} was clicked`);
  });

  return button;
}

A classic pitfall: closures inside loops#

This is one of the most common closure-related bugs, and a favorite interview question. What does this log?

JavaScript
for (var i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i);
  }, 100);
}
// 3
// 3
// 3   <- probably not what you expected

Because var is function-scoped (see Scope & Hoisting), there's only one i shared by all three callbacks. By the time the timers fire, the loop has already finished and i is 3. All three closures point at the same variable, and it now holds its final value.

The fix is almost embarrassingly simple: use let.

JavaScript
for (let i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i);
  }, 100);
}
// 0
// 1
// 2

Because let is block-scoped, JavaScript creates a fresh i for each pass through the loop. Each closure captures its own separate copy instead of sharing one variable.

Tip

This is one of the strongest real-world arguments for preferring let over var: it makes closures inside loops behave the way most people intuitively expect.

Key Takeaways

  • A closure is a function plus a live reference to the variables from the scope where it was created.
  • Closures keep outer variables alive after the outer function has returned — this is normal, not a memory leak, as long as it's intentional.
  • Closures are how JavaScript implements private state, function factories, and callbacks that need to remember context.
  • Looping with var shares one variable across every closure created in the loop; let gives each iteration its own copy.
