Object-Oriented JS

Prototypes & Inheritance

Underneath the friendly class syntax you'll usually write, JavaScript's object model works completely differently from Java or C++. Objects don't inherit from classes — they link directly to other objects. This is prototypal inheritance, and understanding it demystifies a lot of "why does JavaScript do that?" moments.

Every object has a prototype#

Almost every object in JavaScript has an internal link to another object, called its prototype. When you access a property, and it's not found directly on the object, JavaScript automatically checks the prototype — and then its prototype, and so on, until it finds the property or runs out of prototypes.

JavaScript
const animal = {
  eats: true,
  walk() {
    console.log("Animal walks");
  },
};

const rabbit = Object.create(animal); // rabbit's prototype is `animal`
rabbit.jumps = true;

rabbit.jumps; // true — own property
rabbit.eats;  // true — not on rabbit itself; found on its prototype, `animal`
rabbit.walk(); // "Animal walks" — methods are inherited the same way

This chain of "if not found here, check the prototype" is called the prototype chain — it's the exact same lookup idea as the scope chain (see Scope & Hoisting), just for object properties instead of variables.

Where the chain ends#

Every plain object's prototype chain eventually reaches Object.prototype, which is where common methods like .toString() or .hasOwnProperty() actually live — not duplicated on every single object.

JavaScript
const obj = {};
obj.toString(); // "[object Object]" — inherited from Object.prototype

Object.getPrototypeOf(obj) === Object.prototype; // true
Object.getPrototypeOf(Object.prototype);          // null — the chain ends here

Constructor functions#

Before the class keyword existed (and still, underneath it today), JavaScript created many similar objects using a regular function called with new, plus a shared .prototype object for methods.

JavaScript
function Person(name) {
  this.name = name; // own property, unique to each instance
}

// Shared across every instance — added once, used everywhere
Person.prototype.greet = function () {
  console.log(`Hi, I'm ${this.name}`);
};

const ava = new Person("Ava");
const noah = new Person("Noah");

ava.greet();  // "Hi, I'm Ava"
noah.greet(); // "Hi, I'm Noah"

ava.greet === noah.greet; // true — it's the exact same function, shared via the prototype

This sharing is the whole point: instead of every object carrying its own copy of every method (wasting memory), all instances share one set of methods through the prototype, and only keep their own unique data (like name) directly on themselves.

Calling a function with new does four things automatically: creates a new empty object, sets that object's prototype to the constructor's .prototype, sets this to the new object for the duration of the function (see The this Keyword), and returns the new object.

instanceof#

instanceof checks whether an object's prototype chain includes a given constructor's .prototype:

JavaScript
ava instanceof Person; // true
ava instanceof Object;  // true — Person.prototype itself inherits from Object.prototype

Why this matters even though you'll mostly write class#

Modern JavaScript almost always uses the class keyword (next chapter) instead of hand-writing constructor functions. But class is syntax sugar — under the hood, it does exactly what's described on this page: a constructor function plus a shared prototype object for methods.

JavaScript
class Person {
  constructor(name) {
    this.name = name;
  }
  greet() {
    console.log(`Hi, I'm ${this.name}`);
  }
}

typeof Person;                       // "function" — classes are functions underneath
Person.prototype.greet;               // the exact greet method, living on the prototype

Knowing this makes class behavior — like why methods are shared across instances, or what instanceof actually checks — predictable instead of magical.

Note

You may see obj.__proto__ used to read or set a prototype directly. It works, but it's a legacy accessor — prefer Object.getPrototypeOf(obj) and Object.create(proto) in real code.

Key Takeaways

  • JavaScript objects link to other objects via a prototype; missing properties are looked up along this prototype chain.
  • The chain for plain objects ends at Object.prototype, which is where inherited methods like .toString() live.
  • Constructor functions plus a shared .prototype object let many instances share methods without duplicating them.
  • The class keyword is syntax sugar over exactly this mechanism — not a different object model.
