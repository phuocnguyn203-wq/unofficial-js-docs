Working with Data

Working with JSON

JSON (JavaScript Object Notation) is a lightweight, text-based data format. It was inspired by JavaScript's object syntax, but it's language-independent — nearly every programming language can read and write it, which is exactly why it's the default format for APIs and config files.

What JSON looks like#

JSON looks almost like a JavaScript object literal, but it's stricter:

JSON
{
  "name": "Ava",
  "age": 29,
  "isAdmin": false,
  "tags": ["admin", "editor"],
  "address": {
    "city": "Tokyo"
  }
}

The rules that make it stricter than a JS object literal:

  • Property names must be double-quoted strings — name alone or 'name' is invalid.
  • No trailing commas after the last item.
  • No comments.
  • Values can only be: strings, numbers, booleans, null, arrays, or other JSON objects — no functions, no undefined, no Symbol.

JSON.stringify — JavaScript value → JSON text#

Use JSON.stringify() whenever you need to send a JavaScript value somewhere that only understands text — an HTTP request body, localStorage, a file.

JavaScript
const user = { name: "Ava", age: 29, isAdmin: false };

JSON.stringify(user);
// '{"name":"Ava","age":29,"isAdmin":false}'  <- this is now a plain string

A third argument adds indentation, which is invaluable for logging or writing readable config files:

JavaScript
JSON.stringify(user, null, 2);
// {
//   "name": "Ava",
//   "age": 29,
//   "isAdmin": false
// }

JSON.parse — JSON text → JavaScript value#

The reverse operation: turn a JSON-formatted string back into a real, usable JavaScript object.

JavaScript
const raw = '{"name":"Ava","age":29}';
const user = JSON.parse(raw);

user.name; // "Ava" — a real object now, not a string

Warning

JSON.parse throws a SyntaxError if the string isn't valid JSON — always wrap it in a try/catch when parsing data you don't fully control (like a server response or user input). See Error Handling & Debugging.

What doesn't survive the round trip#

Because JSON is a smaller format than JavaScript's full type system, some things are dropped or transformed silently by JSON.stringify:

JavaScript
const data = {
  name: "Ava",
  greet: function () { },   // functions are dropped entirely
  missing: undefined,        // undefined properties are dropped entirely
  when: new Date(),          // Date objects become ISO date strings
  id: Symbol("id"),          // symbols are dropped entirely
};

JSON.stringify(data);
// '{"name":"Ava","when":"2026-07-19T00:00:00.000Z"}'

This is worth remembering: if you serialize an object and parse it back, you get a plain data copy — no functions, no Date instances (just strings that look like dates), no class instances with their prototype intact (see Prototypes & Inheritance).

A common use case: persisting data#

localStorage can only store strings, so JSON is the standard way to save structured data in the browser:

JavaScript
const cart = [{ item: "Book", price: 15 }];

localStorage.setItem("cart", JSON.stringify(cart));

const restored = JSON.parse(localStorage.getItem("cart"));
// restored is a real array again, equal in content to the original cart

This exact pattern — stringify to save, parse to restore — is covered in context in Fetch, Forms & Storage, alongside sending JSON to a server with fetch.

Key Takeaways

  • JSON is a text format for data — it looks like a JS object literal but requires double-quoted keys and allows only a limited set of value types.
  • JSON.stringify() turns a JS value into a JSON string; JSON.parse() turns it back.
  • Functions, undefined, and symbols are silently dropped during stringify; Date objects become plain strings.
  • Always try/catch around JSON.parse() for data you don't fully trust — malformed JSON throws.
