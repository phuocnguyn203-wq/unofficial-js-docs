Language Basics

Variables & Data Types

Variables are named boxes for values. JavaScript decides what kind of value is inside the box at runtime, not when you declare it — that single fact shapes a lot of how the language feels.

Declaring variables: var, let, const#

JavaScript has three ways to declare a variable. In modern code, you'll reach for const by default, let when you need to reassign, and almost never var.

JavaScript
const name = "Ava";     // cannot be reassigned
let age = 29;           // can be reassigned
age = 30;               // OK

var legacy = "old style"; // works, but avoid in new code
KeywordReassignable?ScopeUse it when…
constNoBlockAlways, unless you know you'll reassign.
letYesBlockThe value needs to change (a counter, an accumulator).
varYesFunctionRarely — mostly seen in older codebases.

const doesn't mean "constant value" — it means "this name can't be reassigned." An object or array declared with const can still have its contents changed:

JavaScript
const user = { name: "Ava" };
user.name = "Nora"; // fine — we're mutating the object, not reassigning `user`
// user = {};       // TypeError — this would be a reassignment

Note

var behaves quite differently from let/const when it comes to scope and hoisting. The full comparison lives in Scope & Hoisting — for now, just default to const/let.

The primitive data types#

JavaScript has seven primitive types — simple, immutable values that aren't objects.

TypeExampleDescription
string"hello"Text, wrapped in quotes or backticks.
number42, 3.14Both integers and decimals — there's only one number type.
booleantrue, falseA binary yes/no value.
undefinedundefinedA variable that's been declared but has no value yet.
nullnullAn intentional "no value," set explicitly by you.
bigint10nIntegers larger than Number can safely represent.
symbolSymbol("id")A guaranteed-unique value, often used as an object key.
JavaScript
let city = "Tokyo";        // string
let population = 13960000; // number
let isCapital = true;      // boolean
let nickname;              // undefined — declared, not assigned
let middleName = null;     // null — explicitly "nothing"

Checking a type: typeof#

The typeof operator returns a string naming a value's type — useful for quick checks and debugging.

JavaScript
typeof "hello";     // "string"
typeof 42;           // "number"
typeof true;         // "boolean"
typeof undefined;    // "undefined"
typeof Symbol();     // "symbol"
typeof 10n;          // "bigint"

typeof null;         // "object"  <- a famous, decades-old bug in the language
typeof {};           // "object"
typeof [];           // "object" (arrays are a kind of object — see below)
typeof function(){}; // "function"

Warning

typeof null returning "object" is a mistake baked into JavaScript since 1995 that can never be fixed without breaking the web. To check for null specifically, compare directly: value === null.

Dynamic typing and type coercion#

A variable isn't locked to a type — you can reassign it to hold something completely different:

JavaScript
let value = 42;
value = "now I'm a string";
value = true;

JavaScript will also automatically convert (coerce) values between types when an operator expects a particular type. This is convenient, but it's also the source of JavaScript's most infamous quirks:

JavaScript
"5" + 1;      // "51"  — number is coerced to a string, then concatenated
"5" - 1;      // 4     — string is coerced to a number for subtraction
"5" * "2";    // 10    — both strings coerced to numbers
1 + true;     // 2     — true becomes 1
1 + null;     // 1     — null becomes 0
1 + undefined;// NaN   — undefined can't become a useful number

The rule of thumb: + prefers strings (if either side is a string, it concatenates), while -, *, and / prefer numbers (they'll try to convert both sides to numbers).

=== vs ==#

Because of coercion, JavaScript has two equality operators:

JavaScript
"5" == 5;   // true  — coerces types before comparing
"5" === 5;  // false — no coercion; different types, so not equal

0 == false;  // true
0 === false; // false

Tip

Use === and !== by default. They compare value and type, which avoids an entire category of confusing bugs. Reach for == only when you deliberately want coercion (rare).

Objects: everything else#

Anything that isn't one of the seven primitives is an object — plain objects, arrays, functions, dates, and more. The key practical difference: primitives are compared and copied by value, while objects are compared and copied by reference.

JavaScript
let a = 5;
let b = a;   // b gets a copy of the value 5
b = 10;
console.log(a); // 5 — unaffected

let obj1 = { count: 5 };
let obj2 = obj1;  // obj2 points to the SAME object as obj1
obj2.count = 10;
console.log(obj1.count); // 10 — both names point to one object

This distinction explains a lot of behavior you'll meet later — especially when passing values into functions. It's covered in depth in Objects Deep Dive and Arrays & Array Methods.

Naming variables#

  • Names can contain letters, digits, $ and _, but can't start with a digit.
  • JavaScript is case-sensitive: age and Age are different variables.
  • The convention is camelCase: firstName, totalPrice, isLoggedIn.
  • Reserved words (let, class, return, etc.) can't be used as variable names.

Key Takeaways

  • Default to const; use let when a value needs to change; avoid var in new code.
  • JavaScript has seven primitive types, plus objects for everything else.
  • typeof checks a value's type, but has one famous quirk: typeof null === "object".
  • JavaScript coerces types automatically with ==, +, -, etc. — use === to compare without coercion.
  • Primitives copy by value; objects (including arrays and functions) copy by reference.
