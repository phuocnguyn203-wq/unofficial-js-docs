Browser & Web APIs

Events & Delegation

The browser fires an event for almost everything that happens on a page — clicks, key presses, form submissions, even the page finishing loading. Listening for these events is how JavaScript turns a static page into an interactive one.

Adding an event listener#

JavaScript
const button = document.querySelector("#save-btn");

button.addEventListener("click", function (event) {
  console.log("Button was clicked!");
});

Some of the events you'll reach for constantly:

EventFires when…
clickAn element is clicked (or tapped).
submitA <form> is submitted.
inputAn input's value changes, on every keystroke.
changeAn input's value changes and loses focus (or for checkboxes/selects, immediately).
keydownA key is pressed.
DOMContentLoadedThe HTML has been fully parsed (fired on document).

The event object#

Every listener receives an event object describing what happened — which element triggered it, what key was pressed, and so on.

JavaScript
const form = document.querySelector("form");

form.addEventListener("submit", function (event) {
  event.preventDefault(); // stop the browser's default full-page reload
  console.log("Submitted by:", event.target); // the element the event fired on
});

event.preventDefault() is one of the most commonly used methods here — it stops the browser's built-in behavior for that event (following a link, submitting a form the traditional way) so your JavaScript can handle it instead.

Event bubbling#

When an event fires on an element, it doesn't just run listeners on that exact element — it "bubbles" upward, triggering listeners on every ancestor too, all the way up to document.

JavaScript
// <div id="outer"><button id="inner">Click me</button></div>

document.querySelector("#outer").addEventListener("click", () => {
  console.log("outer div handler ran");
});
document.querySelector("#inner").addEventListener("click", () => {
  console.log("inner button handler ran");
});

// Clicking the button logs both, in this order:
// "inner button handler ran"
// "outer div handler ran"

Call event.stopPropagation() inside a handler if you specifically need to stop this upward bubbling — but it's needed far less often than you'd expect.

Event delegation#

Bubbling isn't just a quirk to work around — it enables a genuinely useful pattern. Instead of attaching a listener to every single item in a list (especially one that changes over time), attach one listener to their shared parent, and check event.target to see which child was actually clicked.

JavaScript
const list = document.querySelector("#todo-list");

list.addEventListener("click", function (event) {
  if (event.target.matches(".delete-btn")) {
    event.target.closest("li").remove();
  }
});

// Works even for <li> items added to the list AFTER this listener was attached —
// there's only ever one listener, on the parent, not one per item.

Tip

Delegation is the standard solution for lists where items are added or removed dynamically. Attaching individual listeners to each item means new items silently have no listener at all — delegating to a stable parent element sidesteps that completely.

Removing a listener#

removeEventListener needs a reference to the exact same function that was added — which means an anonymous inline function can never be removed. Use a named function if you'll need to remove it later.

JavaScript
function handleClick() {
  console.log("clicked");
}

button.addEventListener("click", handleClick);
button.removeEventListener("click", handleClick); // works — same function reference

Key Takeaways

  • addEventListener(type, handler) reacts to browser events; the handler receives an event object describing what happened.
  • event.preventDefault() cancels a default browser behavior (like a form's page reload); event.stopPropagation() stops bubbling.
  • Events bubble from the target element up through every ancestor to document.
  • Event delegation — one listener on a parent, checking event.target — handles dynamic lists better than per-item listeners.
