Browser & Web APIs

Fetch, Forms & Storage

This page ties together everything from the async and DOM chapters into the three things almost every web app needs: talking to a server, reading what a user typed, and remembering data between visits.

The Fetch API#

fetch() makes an HTTP request and returns a promise — a perfect match for async/await from the previous section.

JavaScript
async function getUsers() {
  const response = await fetch("https://api.example.com/users");
  const users = await response.json(); // parses the JSON response body
  return users;
}

By default fetch sends a GET request. For anything else, pass a second options object:

JavaScript
async function createUser(user) {
  const response = await fetch("https://api.example.com/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(user), // see Working with JSON
  });
  return response.json();
}

A critical gotcha: fetch doesn't reject on HTTP errors#

This trips up almost everyone at least once: fetch()'s promise only rejects on a genuine network failure (no connection, DNS failure). A 404 Not Found or a 500 Server Error still counts as a "successful" fetch as far as the promise is concerned — you have to check response.ok yourself.

JavaScript
async function getUser(id) {
  const response = await fetch(`/api/users/${id}`);

  if (!response.ok) {
    // response.ok is true only for status codes 200-299
    throw new Error(`Request failed with status ${response.status}`);
  }

  return response.json();
}

Wrap the call in try/catch (see Error Handling & Debugging) to handle both the network-failure case and the manually-thrown HTTP-error case in one place.

Working with forms#

The standard pattern: prevent the browser's default full-page submission, read the form's values, then send them yourself with fetch.

HTML
<form id="signup-form">
  <input name="email" type="email">
  <button type="submit">Sign up</button>
</form>
JavaScript
const form = document.querySelector("#signup-form");

form.addEventListener("submit", async function (event) {
  event.preventDefault();

  const formData = new FormData(form);   // reads every named field automatically
  const email = formData.get("email");

  const response = await fetch("/api/signup", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ email }),
  });

  if (response.ok) {
    console.log("Signed up!");
  }
});

FormData reads every field with a name attribute automatically, so you don't have to manually query and read each input one by one.

Using localStorage#

localStorage only stores strings, so anything structured needs to go through JSON.stringify/JSON.parse (see Working with JSON):

JavaScript
// Save
localStorage.setItem("theme", "dark");
localStorage.setItem("cart", JSON.stringify([{ item: "Book", price: 15 }]));

// Read
localStorage.getItem("theme");                      // "dark"
JSON.parse(localStorage.getItem("cart"));            // the real array back

// Remove
localStorage.removeItem("theme");
localStorage.clear(); // removes everything this site stored

Note

localStorage is synchronous and per-origin (per domain) — data saved on one site is invisible to another. It's great for preferences and small caches, but not a substitute for a real database for anything sensitive or large.

Key Takeaways

  • fetch() returns a promise that resolves for both successful and failed HTTP responses — always check response.ok yourself.
  • FormData reads every named form field at once, instead of querying each input individually.
  • localStorage persists indefinitely and isn't sent to the server; sessionStorage clears when the tab closes; cookies are small and sent with every request.
  • Browser storage only holds strings — use JSON.stringify/JSON.parse to store structured data.
