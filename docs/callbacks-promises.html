Asynchronous JavaScript

Callbacks & Promises

Now that you know why JavaScript needs to handle things asynchronously (see The Event Loop), this page covers the two patterns for actually writing async code: the original callback pattern, and the promise that replaced it for most use cases.

What is a callback?#

A callback is simply a function passed into another function, to be called later — usually once some operation finishes. You've already used this idea with array methods (see Arrays & Array Methods); async callbacks are the same concept, just triggered by time or I/O instead of a loop.

JavaScript
function fetchUserName(id, callback) {
  setTimeout(() => {
    // pretend this took time — a network request, a database lookup
    callback({ id, name: "Ava" });
  }, 1000);
}

fetchUserName(1, function (user) {
  console.log(`Got user: ${user.name}`);
});
console.log("Fetching..."); // logs first — fetchUserName doesn't block

The problem: callback hell#

Callbacks work fine for one async step. The trouble starts when several async steps depend on each other — each one has to be nested inside the previous one's callback, producing a growing pyramid of indentation that's hard to read and even harder to add error handling to.

JavaScript
getUser(1, function (user) {
  getPosts(user.id, function (posts) {
    getComments(posts[0].id, function (comments) {
      console.log(comments);
      // and it keeps going, one level deeper each time...
    }, handleError);
  }, handleError);
}, handleError);

This shape earned its own nickname: callback hell (or "the pyramid of doom"). Promises were introduced in ES6 specifically to fix this.

What is a Promise?#

A promise is an object representing a value that isn't available yet, but will be at some point — either successfully (fulfilled) or with an error (rejected). Every promise is in exactly one of three states:

  • pending — still working, no result yet.
  • fulfilled — completed successfully, with a value.
  • rejected — failed, with a reason (usually an Error).

Once a promise settles (fulfills or rejects), it stays that way forever — it can't change state again.

Creating a promise#

Most of the time you'll consume promises returned by built-in functions like fetch(), not create your own. But knowing how they're built removes the mystery:

JavaScript
function fetchUserName(id) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (id > 0) {
        resolve({ id, name: "Ava" }); // success
      } else {
        reject(new Error("Invalid id")); // failure
      }
    }, 1000);
  });
}

The function passed to new Promise() is called the executor — it runs immediately, and receives two functions: call resolve(value) on success, or reject(error) on failure.

Consuming a promise: then, catch, finally#

JavaScript
fetchUserName(1)
  .then(user => console.log(`Got user: ${user.name}`))
  .catch(error => console.error("Failed:", error.message))
  .finally(() => console.log("Done, either way"));

.then() runs when the promise fulfills; .catch() runs when it rejects (or when an earlier .then() throws); .finally() runs regardless of the outcome — perfect for cleanup like hiding a loading spinner.

Watch the timing: step through that exact chain and see when each callback actually runs — including why .catch gets skipped entirely on the success path.

Chaining: solving callback hell#

The real payoff: each .then() returns a new promise, so you can chain steps in a flat, readable sequence instead of nesting them.

JavaScript
getUser(1)
  .then(user => getPosts(user.id))
  .then(posts => getComments(posts[0].id))
  .then(comments => console.log(comments))
  .catch(error => console.error("Something failed:", error.message));

One .catch() at the end handles a failure from any step in the chain — if getUser, getPosts, or getComments rejects, execution skips straight to it. That single flat chain replaces the entire nested pyramid from before.

Running promises together#

Four static methods handle common multi-promise situations:

MethodResolves when…Use it for
Promise.all()every promise fulfills (or any one rejects, immediately)Run several independent requests in parallel, need all results.
Promise.allSettled()every promise finishes, fulfilled or rejectedRun several requests, and you want every result even if some fail.
Promise.race()the first promise to settle, winsA timeout race, or "whichever finishes first."
Promise.any()the first promise to fulfill (ignores rejections unless all reject)Try several sources, take the first success.
JavaScript
const requests = [fetchUser(1), fetchUser(2), fetchUser(3)];

Promise.all(requests)
  .then(users => console.log("All loaded:", users))
  .catch(error => console.error("At least one failed:", error));

Promise.allSettled(requests).then(results => {
  results.forEach(result => {
    if (result.status === "fulfilled") console.log(result.value);
    else console.log("failed:", result.reason);
  });
});

Tip

Promise.all() is fast but fragile — one rejection cancels the whole batch. Promise.allSettled() is slower to react but never short-circuits, which is usually what you want when the requests are independent of each other.

Key Takeaways

  • A callback is a function invoked later; deeply nested callbacks for sequential async steps produce hard-to-read "callback hell."
  • A promise represents a future value with exactly one of three states: pending, fulfilled, or rejected — and it settles only once.
  • .then()/.catch()/.finally() consume a promise; chaining .then() calls replaces nested callbacks with a flat sequence.
  • Promise.all, .allSettled, .race, and .any each combine multiple promises with different failure behavior.
