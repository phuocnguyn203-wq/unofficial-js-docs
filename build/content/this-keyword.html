Scope, Closures & this

The this Keyword

this is probably the single most confusing keyword in JavaScript — not because the rule is complicated, but because it's determined by how a function is called, not where it's written. Learn the call-site rule once, and most of the confusion disappears.

The core rule

Forget any assumptions from other languages. In a regular JavaScript function, this is decided fresh, every time, by looking at the call site — the exact way the function was invoked. The same function can have a different this every time it's called.

[[code lang="js"]] function whoAmI() { console.log(this); } whoAmI(); // called plainly const obj = { whoAmI }; obj.whoAmI(); // called as a method [[/code]]

Same function, two very different values of this — because it was called differently each time. There are four patterns that cover almost every case.

1. Plain function call

Calling a function with no context gives this as undefined in strict mode (including inside classes and modules), or the global object in old-style non-strict code.

[[code lang="js"]] "use strict"; function show() { console.log(this); } show(); // undefined [[/code]]

2. Method call (implicit binding)

When a function is called as a property of an object, this is that object.

[[code lang="js"]] const user = { name: "Ava", greet() { console.log(`Hi, I'm ${this.name}`); }, }; user.greet(); // "Hi, I'm Ava" — this === user [[/code]]

3. Explicit binding: call, apply, bind

You can set this yourself using three built-in function methods:

[[code lang="js"]] function greet() { console.log(`Hi, I'm ${this.name}`); } const ava = { name: "Ava" }; const noah = { name: "Noah" }; greet.call(ava); // "Hi, I'm Ava" — calls immediately, this = ava greet.apply(noah); // "Hi, I'm Noah" — same as call, but takes args as an array const greetAva = greet.bind(ava); // returns a NEW function, permanently bound to ava greetAva(); // "Hi, I'm Ava" — can be called anytime, this never changes [[/code]]

call and apply invoke the function right away with a chosen this — they only differ in how they pass extra arguments (call(thisArg, a, b) vs apply(thisArg, [a, b])). bind doesn't call the function at all — it returns a new function permanently locked to the this you gave it, which is perfect for callbacks that will be invoked later by someone else's code.

4. new binding (constructors)

When a function is called with new, JavaScript creates a brand-new object and sets this to it inside the function.

[[code lang="js"]] function Person(name) { this.name = name; // this === the new object being constructed } const ava = new Person("Ava"); ava.name; // "Ava" [[/code]]

This is the mechanism the class syntax builds on top of — see Classes.

Arrow functions: no binding of their own

Arrow functions break the call-site rule on purpose. They don't have their own this at all — they capture this lexically, from the scope they were defined in, exactly like a closure captures a variable.

[[code lang="js"]] const user = { name: "Ava", regularGreet: function () { console.log(this.name); // "Ava" — this is user, called as a method }, arrowGreet: () => { console.log(this.name); // undefined — arrow ignores the call site entirely }, }; user.regularGreet(); // "Ava" user.arrowGreet(); // undefined — `this` here came from the outer (module/global) scope [[/code]]

This makes arrow functions the wrong choice for object methods that need this — but the right choice for callbacks nested inside a method, where you want to keep the surrounding this.

The classic pitfall: losing this

This is the single most common this-related bug. It happens whenever a method gets separated from its object — passed as a callback, assigned to a variable, and so on.

[[code lang="js"]] const user = { name: "Ava", greet() { console.log(`Hi, I'm ${this.name}`); }, }; setTimeout(user.greet, 1000); // "Hi, I'm undefined" — greet was extracted and called plainly, // losing its connection to `user` entirely [[/code]]

Two common fixes:

[[code lang="js"]] // Fix 1: bind it setTimeout(user.greet.bind(user), 1000); // Fix 2: wrap it in an arrow function, which captures `this` from here setTimeout(() => user.greet(), 1000); [[/code]]

Tip

When you see this.something unexpectedly undefined, the first thing to check is how the function was called — was it detached from its object at some point? That's almost always the cause.

Precedence, if you need it

When more than one rule could apply, this is the order of priority: new binding wins first, then explicit binding (call/apply/bind), then implicit (method call), and plain function calls come last. Arrow functions ignore all of this and always use their lexical this.

Key Takeaways

  • this is set by how a function is called (the call site), not where it's defined.
  • Method calls set this to the object before the dot; plain calls give undefined in strict mode.
  • call/apply invoke a function with a chosen this immediately; bind returns a new function permanently bound to one.
  • Arrow functions have no this of their own — they inherit it lexically from their surrounding scope, which is why they're safe for nested callbacks but wrong for object methods.
