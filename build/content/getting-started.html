Get Started

Your First Script

You don't need to install anything to start writing JavaScript. This page covers the three places you'll run it while learning: the browser console, a linked script file, and (briefly) Node.js.

The browser console

Every browser ships a set of developer tools with a JavaScript console built in. It's the fastest way to try something out.

  • Chrome / Edge: F12 or Ctrl+Shift+J, then the Console tab.
  • Firefox: F12, then the Console tab.
  • Safari: enable the Develop menu in Preferences, then Cmd+Option+C.

Open one now and type this directly into it:

[[code lang="js"]] console.log("Hello, JavaScript!"); [[/code]] [[code lang="console"]] Hello, JavaScript! [[/code]]

console.log() prints a value to the console. You'll use it constantly to inspect what your code is doing — it's the most common debugging tool in JavaScript.

Linking JavaScript to a web page

In a real project, JavaScript lives in its own .js file and gets linked into an HTML page with a <script> tag.

[[code lang="html"]] My Page

Hello!

[[/code]] [[code lang="js"]] // script.js console.log("script.js has loaded"); document.querySelector("h1").style.color = "tomato"; [[/code]]

Two details matter here:

  • Where the script tag goes. The browser reads HTML top to bottom. If your script runs before the rest of the page exists, it can't find elements like <h1> yet. Putting the script at the end of <body>, or using the defer attribute, avoids this.
  • defer vs async. defer downloads the script in the background and runs it after the page is fully parsed, in order. async downloads in the background too, but runs the script the instant it's ready — possibly before the page finishes parsing, and not necessarily in order relative to other scripts. For most page scripts, defer is the safer default.

Tip

You can also skip the file entirely for quick experiments with an inline <script>…</script> block directly in your HTML. It's fine for learning, but real projects almost always use separate .js files.

Strict mode

Add "use strict"; at the top of a script or function to opt into a stricter variant of JavaScript that catches common mistakes — like accidentally creating a global variable by forgetting let — and turns them into errors instead of silent bugs.

[[code lang="js"]] "use strict"; function example() { x = 10; // ReferenceError: x is not defined (without strict mode, this // would silently create a global variable named x) } [[/code]]

Modern JavaScript modules (see Modules) and classes are automatically in strict mode — you'll increasingly write strict-mode code without ever typing the directive yourself.

Comments

Comments are ignored by the engine and exist purely for humans reading the code later (including future you).

[[code lang="js"]] // A single-line comment /* * A multi-line comment, * often used for short explanations above a function. */ function add(a, b) { return a + b; // inline comments work too } [[/code]]

A quick look at Node.js

Everything in this guide runs in the browser, but it's worth knowing that Node.js lets you run the exact same JavaScript language outside the browser — on a server, in a terminal, in build tools. If you have Node installed, you can run a file directly:

[[code lang="bash"]] node script.js [[/code]]

Node adds its own APIs for things browsers don't need (reading files, starting servers), while removing browser-only APIs like document. The core language — variables, functions, closures, promises — is identical either way, which is exactly why this guide's syntax and concept chapters apply regardless of where you eventually run your code.

Key Takeaways

  • The browser console is the fastest way to experiment — open devtools and start typing.
  • Real projects link JavaScript via <script src="..." defer>, usually placed so it runs after the page has parsed.
  • "use strict" catches common mistakes early; modules and classes use it automatically.
  • Node.js runs the same JavaScript language outside the browser, with different built-in APIs.
