Scope, Closures & this

Scope & Hoisting

Scope answers one question: from any given line of code, which variables can you see? Hoisting answers another: when does a variable start existing? Together they explain some of JavaScript's most confusing-looking behavior — and they're the foundation for closures, the next chapter.

What is scope?

Scope is the region of code where a variable is accessible. JavaScript has three kinds:

  • Global scope — declared outside any function or block; accessible everywhere.
  • Function scope — declared inside a function; accessible only within it (and functions nested inside it).
  • Block scope — declared inside { } (an if, a for loop, or a bare block); accessible only within that block.
[[code lang="js"]] const globalVar = "I'm everywhere"; function outer() { const functionVar = "I only exist inside outer()"; if (true) { const blockVar = "I only exist inside this if-block"; console.log(globalVar, functionVar, blockVar); // all visible here } console.log(blockVar); // ReferenceError: blockVar is not defined } [[/code]]

var is function-scoped; let/const are block-scoped

This is the single biggest practical difference between var and let/const, and the main reason modern code avoids var.

[[code lang="js"]] if (true) { var leaked = "I escape the block"; let contained = "I stay inside the block"; } console.log(leaked); // "I escape the block" — var ignores block boundaries console.log(contained); // ReferenceError: contained is not defined [[/code]]

var only respects function boundaries, not block boundaries — so a var declared inside an if or a for loop "leaks" into the surrounding function. This has caused real bugs for decades, which is exactly why let and const were introduced in ES6.

Hoisting

Before running your code, the JavaScript engine scans the current scope and registers every declaration it finds — as if it "hoists" them to the top. But what gets hoisted, and in what state, differs by keyword:

[[code lang="js"]] console.log(hoistedVar); // undefined (not an error!) var hoistedVar = "value"; sayHi(); // "Hi!" — works fine function sayHi() { console.log("Hi!"); } [[/code]]

var declarations are hoisted and initialized to undefined immediately — so reading one before its assignment gives undefined rather than an error. Function declarations are hoisted completely, body and all, so you can call them before they appear in the file.

The Temporal Dead Zone (TDZ)

let and const are hoisted too, technically — but they're not initialized. Accessing them before their declaration line throws an error instead of silently giving undefined. The span between the start of the scope and the actual declaration is called the temporal dead zone.

[[code lang="js"]] console.log(value); // ReferenceError: Cannot access 'value' before initialization let value = "hello"; [[/code]]

Tip

The TDZ isn't a design flaw — it's a safety net. It turns a whole class of "used a variable before it had a sensible value" bugs into loud, immediate errors instead of silent undefineds.

Function expressions are not hoisted like declarations

Only function declarations get the full hoisting treatment. A function stored in a const/let/var follows that variable's own hoisting rules instead:

[[code lang="js"]] sayHi(); // TypeError: sayHi is not a function (it's undefined at this point) var sayHi = function () { console.log("Hi!"); }; [[/code]]

The scope chain

When JavaScript looks up a variable, it checks the current scope first, then the scope it's nested inside, then that scope's parent, and so on out to the global scope. This chain of lookups is called the scope chain.

[[code lang="js"]] const level1 = "top"; function outer() { const level2 = "middle"; function inner() { const level3 = "bottom"; // inner() can see all three, by walking up the scope chain console.log(level1, level2, level3); } inner(); } outer(); // "top middle bottom" [[/code]]

Notice the lookup only goes outward — outer() can't see level3, because inner()'s scope is nested inside outer(), not the other way around.

This "remembering the scope a function was created in" is exactly the mechanism behind closures — covered next.

Key Takeaways

  • JavaScript has global, function, and block scope; let/const respect block scope, var does not.
  • Declarations are hoisted: var becomes undefined until assigned; function declarations are fully usable immediately.
  • let/const are hoisted but unusable until their declaration line runs — the Temporal Dead Zone.
  • Variable lookups walk outward through the scope chain: current scope → enclosing scope → ... → global.
