Functional Programming

Functional programming isn't a separate language feature — it's a style, built entirely on ideas you've already met: functions as values (see Functions) and non-mutating array methods (see Arrays & Array Methods). This page names the underlying principles so you can apply them deliberately.

Pure functions

A function is pure if it always returns the same output for the same input, and causes no observable side effects (no modifying external variables, no logging, no network calls).

[[code lang="js"]] // Impure — depends on and mutates external state let total = 0; function addToTotal(amount) { total += amount; // side effect: modifies a variable outside itself return total; } // Pure — same input always gives the same output, nothing outside is touched function add(a, b) { return a + b; } [[/code]]

Pure functions are easier to test (no setup or mocking required), easier to reason about (you can understand one in isolation), and safe to run in any order — which matters a lot once code becomes concurrent or gets cached/memoized.

Immutability

Rather than modifying data in place, functional style creates and returns new data, leaving the original untouched — the same principle behind map/filter and object/array spreading.

[[code lang="js"]] // Mutating — changes the original array function addItemMutating(cart, item) { cart.push(item); return cart; } // Immutable — returns a new array, original is untouched function addItem(cart, item) { return [...cart, item]; } const cart = ["apple"]; const newCart = addItem(cart, "banana"); cart; // ["apple"] — unchanged newCart; // ["apple", "banana"] [[/code]]

This matters most when several parts of a program hold a reference to the same object (see Variables & Data Types) — immutability guarantees nobody else's copy changes out from under them.

Function composition

Building a complex operation by combining small, single-purpose functions, where each one's output feeds the next one's input.

[[code lang="js"]] const trim = str => str.trim(); const lowercase = str => str.toLowerCase(); const removeSpaces = str => str.replace(/\s+/g, "-"); function slugify(str) { return removeSpaces(lowercase(trim(str))); } slugify(" Hello World "); // "hello-world" [[/code]]

Each helper does exactly one thing and can be tested and reused independently — lowercase and trim are obviously useful elsewhere too, unlike one large function that did all three steps inline.

Currying

Currying transforms a function that takes multiple arguments into a chain of functions that each take one argument — enabling the function-factory pattern from Closures, formalized.

[[code lang="js"]] // Normal function function add(a, b) { return a + b; } // Curried version function curriedAdd(a) { return function (b) { return a + b; }; } curriedAdd(2)(3); // 5 const add5 = curriedAdd(5); // a specialized function, "waiting" for one more argument add5(10); // 15 [[/code]]

This is most useful for pre-configuring a general function into a specific one, like multiplyBy(2) from the closures chapter — currying is that same pattern, just applied consistently.

Why this matters

You don't need to write 100% pure, immutable code everywhere — JavaScript isn't a purely functional language, and plenty of real-world code (DOM manipulation, for instance) is inherently about side effects. But leaning on these ideas where you can — especially pure functions for data transformations — tends to produce code that's easier to test, easier to debug (no hidden state to track down), and safer to reuse.

Key Takeaways

  • A pure function returns the same output for the same input and has no side effects — easier to test and reason about.
  • Prefer returning new data over mutating existing data, especially for values shared across your program.
  • Function composition builds complex behavior from small, single-purpose, independently testable functions.
  • Currying turns a multi-argument function into a chain of single-argument functions, useful for pre-configuring specialized versions.
