Modern JavaScript (ES6+)

Modules

As a project grows past one script file, you need a way to split code up while still letting different files share functions and values safely. ES6 modules are the standardized answer, replacing a decade of ad-hoc workarounds.

The problem: one shared global scope

Before modules, every <script> tag on a page shared one single global scope. Two scripts that happened to declare a variable with the same name would silently clash and overwrite each other.

[[code lang="html"]] [[/code]] [[code lang="js"]] // a.js var user = "Ava"; // b.js var user = "Noah"; // silently overwrites a.js's variable — same global scope [[/code]]

Modules fix this by giving every file its own scope. Nothing is shared between files unless it's explicitly exported and imported.

Exporting

A module shares values using export. A file can have any number of named exports, plus at most one default export.

[[code lang="js"]] // math.js export const PI = 3.14159; export function add(a, b) { return a + b; } export default function multiply(a, b) { // the default export return a * b; } [[/code]]

Importing

[[code lang="js"]] // app.js import multiply, { PI, add } from "./math.js"; add(2, 3); // 5 multiply(2, 3); // 6 PI; // 3.14159 [[/code]]

Named exports are imported with { } and must match the exported name exactly (though they can be renamed — see below). A default export is imported without braces, and the importing file can call it whatever it wants.

Using modules in the browser

Add type="module" to your script tag to opt into module behavior:

[[code lang="html"]] [[/code]]

Module scripts get a few automatic behaviors worth knowing:

  • They always run in strict mode.
  • They're deferred automatically — no need for the defer attribute (see Your First Script).
  • Top-level await is allowed directly in a module, outside any async function.
  • They must be served over http(s):// — opening an HTML file directly from disk (file://) will block module imports due to browser security restrictions. Use a local server (or just deploy — GitHub Pages serves everything over https).

Renaming on import or export

Use as to rename something, which is handy for avoiding naming collisions between files:

[[code lang="js"]] export { add as sum }; import { sum as addNumbers } from "./math.js"; addNumbers(2, 3); // 5 [[/code]]

Importing everything as a namespace

[[code lang="js"]] import * as MathUtils from "./math.js"; MathUtils.add(2, 3); MathUtils.PI; [[/code]]

Tip

Prefer named exports for most things — they make it clear at the import line exactly what a file provides, and tools can catch typos. Reserve a default export for a file that has one obvious "main" thing to offer, like a single class or component.

Key Takeaways

  • Each module file has its own scope — nothing leaks into the global scope unless explicitly exported and imported.
  • export/export default share values; import brings them into another file, optionally renamed with as.
  • Browser module scripts (type="module") are strict-mode and deferred by default, and require serving over http(s), not opened directly from disk.
