Errors & Debugging

Error Handling & Debugging

Things go wrong: a server times out, a user submits bad data, a value is unexpectedly null. This page covers handling failure gracefully instead of letting it crash your app, plus the everyday tools for tracking bugs down in the first place.

try / catch / finally

try runs code that might fail; if it throws, execution jumps straight to catch instead of crashing the program. finally runs regardless of whether an error occurred.

[[code lang="js"]] try { const data = JSON.parse("{ invalid json"); console.log(data); } catch (error) { console.error("Failed to parse:", error.message); } finally { console.log("This always runs, error or not."); } [[/code]]

Anything that throws inside the try block — a bad function call, an intentional throw, a rejected await (see Async/Await) — is caught. Code after the failure point inside try does not run.

The Error object

JavaScript's built-in errors all share the same basic shape: a message describing what went wrong, a name identifying the error type, and a stack trace for debugging.

Error typeThrown when…
TypeErrorAn operation is performed on the wrong type — calling a non-function, reading a property of null.
RangeErrorA number is outside an allowed range — like an invalid array length.
ReferenceErrorA variable doesn't exist, or is accessed in its temporal dead zone.
SyntaxErrorInvalid syntax — including from JSON.parse on malformed JSON.
ErrorThe generic base type — also what you'll throw yourself, most of the time.
[[code lang="js"]] try { null.name; } catch (error) { error instanceof TypeError; // true error.message; // "Cannot read properties of null (reading 'name')" } [[/code]]

Throwing your own errors

Use throw with a real Error object (not a plain string) so callers get a consistent shape to inspect, including a stack trace.

[[code lang="js"]] function withdraw(balance, amount) { if (amount > balance) { throw new Error("Insufficient funds"); } return balance - amount; } try { withdraw(100, 150); } catch (error) { console.error(error.message); // "Insufficient funds" } [[/code]]

Custom error types

For larger apps, extending the built-in Error class (see Classes) lets you distinguish different failure kinds with instanceof, instead of parsing message strings.

[[code lang="js"]] class ValidationError extends Error { constructor(message, field) { super(message); this.name = "ValidationError"; this.field = field; } } function setAge(age) { if (age < 0) { throw new ValidationError("Age cannot be negative", "age"); } } try { setAge(-5); } catch (error) { if (error instanceof ValidationError) { console.error(`Invalid ${error.field}: ${error.message}`); } else { throw error; // not ours to handle — let it propagate } } [[/code]]

The principle: only catch what you can act on

The syntax is easy; knowing when to reach for it is what actually matters in a real codebase. The most common mistake isn't forgetting a try block — it's wrapping everything in one "just in case." A catch block should only exist if it does one of three things: recover (retry, fall back to a default), translate (turn a low-level failure into one the caller can understand), or clean up (release a lock, close a connection) before letting the error continue on. If a catch block does none of these, delete it and let the error propagate — some other layer up the call stack is better positioned to deal with it.

[[code lang="js"]] // Bad: catches an error it can't do anything about, then hides it. // The caller has no idea the save failed — it just gets undefined back. async function saveOrder(order) { try { return await db.orders.insert(order); } catch (error) { console.log(error); } } // Good: nothing to add here, so don't catch at all — let it bubble up // to a layer that actually knows what to do with a failed save. async function saveOrder(order) { return await db.orders.insert(order); } [[/code]]

The empty catch block

A catch block that logs and swallows an error is barely better than one that does nothing at all — the program keeps running in a state the rest of the code assumes is fine, and the failure resurfaces later, somewhere harder to trace back to the real cause. If you can't meaningfully handle an error at a given point, don't catch it there.

Operational errors vs. programmer errors

Deciding whether to reach for try/catch at all starts with naming what kind of failure you're looking at.

KindExamplesWhat to do
Operational — expected, not a bugInvalid user input, a duplicate email on signup, a timed-out network request, a file that doesn't exist yet.Anticipate it. Catch it, and turn it into a clear response, a retry, or a fallback.
Programmer — a bugCalling a function that doesn't exist, passing a string where an object was expected, an off-by-one that reads past an array.Don't catch and paper over it. Let it crash loudly (or hit a top-level handler that logs and alerts) so it gets fixed at the source.

The reason this distinction matters: catching a programmer error and handling it like an operational one doesn't fix the bug — it just hides it until it fails again later, in a more confusing way, further from where it actually went wrong. A try/catch around code that reads user.address.city won't fix a missing address; it'll just mask the real bug (the caller forgot to validate the shape of user) behind a generic error message.

Where to catch: pick a boundary, not every layer

In any app with more than one layer — say, a typical 3-tier backend split into routes/controllers, services, and a data-access/repository layer — the same question comes up constantly: which layer should have the try/catch? The answer follows directly from the principle above: catch at whichever layer has enough context to actually do something, and let every other layer stay clean.

Repository

Talks to the database. Only this layer understands driver-specific errors (like a Postgres unique-violation code) — so it's the one place that translates them into a domain error.

Service

Enforces business rules. Catches only the specific domain errors it can act on; everything else passes straight through untouched.

Controller

Handles the HTTP request. No manual try/catch per route — errors are forwarded to one centralized handler.

an uncaught error keeps propagating up, unchanged, until something can act on it
Rule of thumb: catch where you have enough context to recover, translate, or clean up — not reflexively at every function call.

Repository layer — translate the errors only you understand

The database driver's error codes are meaningless outside this layer. Catch them here, where you know what they mean, and convert them into an error the rest of the app can reason about. Preserve the original with the cause option so nothing is lost for debugging.

[[code lang="js"]] // user.repository.js async function create({ email, passwordHash }) { try { return await db.query( "INSERT INTO users (email, password_hash) VALUES ($1, $2) RETURNING *", [email, passwordHash] ); } catch (error) { if (error.code === "23505") { // Postgres unique_violation — only this layer knows what that code means throw new DuplicateEmailError(email, { cause: error }); } throw error; // anything else isn't ours to interpret — let it propagate as-is } } [[/code]]

Service layer — enforce rules, add domain meaning

The service layer works entirely in terms of domain errors, not database errors. It catches only the ones it has a rule for; anything unrecognized passes straight through.

[[code lang="js"]] // user.service.js async function registerUser({ email, password }) { if (!email.includes("@")) { throw new ValidationError("A valid email is required", "email"); } try { return await userRepository.create({ email, passwordHash: hash(password) }); } catch (error) { if (error instanceof DuplicateEmailError) { throw new ConflictError(`An account with ${email} already exists`, { cause: error }); } throw error; // not a case this layer knows how to handle } } [[/code]]

Controller layer — one centralized boundary

Routes shouldn't each carry their own try/catch — that scatters the same "what status code does this error mean" logic across every handler. Instead, wrap route handlers once so a thrown or rejected error is forwarded to a single error-handling middleware, which is the one real decision point for turning an error into an HTTP response.

[[code lang="js"]] // asyncHandler.js — forwards a thrown or rejected error to Express instead of crashing the process const asyncHandler = (fn) => (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next); // user.controller.js router.post("/users", asyncHandler(async (req, res) => { const user = await userService.registerUser(req.body); res.status(201).json(user); })); // error-handler.js — registered last; the only place that decides what the client sees app.use((error, req, res, next) => { if (error instanceof ValidationError) { return res.status(400).json({ error: error.message, field: error.field }); } if (error instanceof ConflictError) { return res.status(409).json({ error: error.message }); } // Not a recognized domain error — treat it as a programmer error: // log the full detail server-side, but never leak internals to the client. console.error(error); res.status(500).json({ error: "Something went wrong" }); }); [[/code]]

Notice what each layer does not do: the repository doesn't know about HTTP, the service doesn't know about Postgres error codes, and the controller doesn't know about either — it just trusts that whatever reaches the error middleware is either a domain error it can map to a status code, or a bug worth logging loudly. That separation is what makes the error handling maintainable as the app grows past a handful of routes.

Why { cause: error }

The cause option (ES2022) lets a higher-level error keep a reference to the lower-level one it was translated from, without exposing it in the message. error.cause is still there in logs and debuggers, so translating an error for the caller's benefit doesn't cost you any information for your own.

Don't use try/catch for control flow

If a "failure" is actually an expected, routine outcome — a key that might not be in a map, a record that might not exist yet — check for it directly (an if, .has(), optional chaining) instead of relying on a throw/catch to route normal logic. Exceptions should signal something exceptional; using them for everyday branching makes the code both slower and harder to follow.

Debugging with the console

console.log() is just the start. A few methods worth knowing:

MethodWhat it does
console.warn()Logs with a warning icon/color — easier to spot.
console.error()Logs as an error, with a full stack trace.
console.table(data)Renders an array of objects as a readable table.
console.group() / groupEnd()Indents a related block of logs together.
console.time(label) / timeEnd(label)Measures how long code between the two calls took.
[[code lang="js"]] console.table([ { name: "Ava", age: 29 }, { name: "Noah", age: 34 }, ]); console.time("fetch"); await fetchData(); console.timeEnd("fetch"); // "fetch: 342.15ms" [[/code]]

Breakpoints and the debugger statement

Console logging is fast, but a real breakpoint lets you pause execution and inspect every variable in scope at that exact moment. In your browser's devtools Sources tab, click a line number to set one — or add this directly in your code:

[[code lang="js"]] function processOrder(order) { debugger; // execution pauses here whenever devtools is open // ... } [[/code]]

Once paused, you can step through code line by line, inspect the current call stack, and evaluate expressions live in the console — usually far faster than adding and removing console.log calls.

Key Takeaways

  • try/catch/finally handles failures gracefully instead of crashing; finally always runs.
  • Built-in errors share a common shape (message, name, stack); TypeError, RangeError, and ReferenceError cover most cases.
  • Extend Error for custom error types you can distinguish with instanceof.
  • Only catch an error if you'll recover, translate, or clean up — otherwise let it propagate; an empty or log-and-swallow catch just hides a bug until it resurfaces later.
  • Separate operational errors (expected — handle them) from programmer errors (bugs — let them fail loudly and fix the cause).
  • In a layered app, catch at the boundary with enough context to act (e.g. repository translates DB errors, service enforces business rules, controller forwards to one centralized error handler) — not reflexively in every function.
  • console.table, console.group, and breakpoints (via debugger or devtools) go far beyond console.log for real debugging.
