Errors & Debugging

Error Handling & Debugging

Things go wrong: a server times out, a user submits bad data, a value is unexpectedly null. This page covers handling failure gracefully instead of letting it crash your app, plus the everyday tools for tracking bugs down in the first place.

try / catch / finally

try runs code that might fail; if it throws, execution jumps straight to catch instead of crashing the program. finally runs regardless of whether an error occurred.

[[code lang="js"]] try { const data = JSON.parse("{ invalid json"); console.log(data); } catch (error) { console.error("Failed to parse:", error.message); } finally { console.log("This always runs, error or not."); } [[/code]]

Anything that throws inside the try block — a bad function call, an intentional throw, a rejected await (see Async/Await) — is caught. Code after the failure point inside try does not run.

The Error object

JavaScript's built-in errors all share the same basic shape: a message describing what went wrong, a name identifying the error type, and a stack trace for debugging.

Error typeThrown when…
TypeErrorAn operation is performed on the wrong type — calling a non-function, reading a property of null.
RangeErrorA number is outside an allowed range — like an invalid array length.
ReferenceErrorA variable doesn't exist, or is accessed in its temporal dead zone.
SyntaxErrorInvalid syntax — including from JSON.parse on malformed JSON.
ErrorThe generic base type — also what you'll throw yourself, most of the time.
[[code lang="js"]] try { null.name; } catch (error) { error instanceof TypeError; // true error.message; // "Cannot read properties of null (reading 'name')" } [[/code]]

Throwing your own errors

Use throw with a real Error object (not a plain string) so callers get a consistent shape to inspect, including a stack trace.

[[code lang="js"]] function withdraw(balance, amount) { if (amount > balance) { throw new Error("Insufficient funds"); } return balance - amount; } try { withdraw(100, 150); } catch (error) { console.error(error.message); // "Insufficient funds" } [[/code]]

Custom error types

For larger apps, extending the built-in Error class (see Classes) lets you distinguish different failure kinds with instanceof, instead of parsing message strings.

[[code lang="js"]] class ValidationError extends Error { constructor(message, field) { super(message); this.name = "ValidationError"; this.field = field; } } function setAge(age) { if (age < 0) { throw new ValidationError("Age cannot be negative", "age"); } } try { setAge(-5); } catch (error) { if (error instanceof ValidationError) { console.error(`Invalid ${error.field}: ${error.message}`); } else { throw error; // not ours to handle — let it propagate } } [[/code]]

Debugging with the console

console.log() is just the start. A few methods worth knowing:

MethodWhat it does
console.warn()Logs with a warning icon/color — easier to spot.
console.error()Logs as an error, with a full stack trace.
console.table(data)Renders an array of objects as a readable table.
console.group() / groupEnd()Indents a related block of logs together.
console.time(label) / timeEnd(label)Measures how long code between the two calls took.
[[code lang="js"]] console.table([ { name: "Ava", age: 29 }, { name: "Noah", age: 34 }, ]); console.time("fetch"); await fetchData(); console.timeEnd("fetch"); // "fetch: 342.15ms" [[/code]]

Breakpoints and the debugger statement

Console logging is fast, but a real breakpoint lets you pause execution and inspect every variable in scope at that exact moment. In your browser's devtools Sources tab, click a line number to set one — or add this directly in your code:

[[code lang="js"]] function processOrder(order) { debugger; // execution pauses here whenever devtools is open // ... } [[/code]]

Once paused, you can step through code line by line, inspect the current call stack, and evaluate expressions live in the console — usually far faster than adding and removing console.log calls.

Key Takeaways

  • try/catch/finally handles failures gracefully instead of crashing; finally always runs.
  • Built-in errors share a common shape (message, name, stack); TypeError, RangeError, and ReferenceError cover most cases.
  • Extend Error for custom error types you can distinguish with instanceof.
  • console.table, console.group, and breakpoints (via debugger or devtools) go far beyond console.log for real debugging.
