Memory & Performance

Most everyday JavaScript never requires thinking about memory directly — the engine handles it. But knowing roughly how it works helps you recognize the handful of situations where it doesn't, and picks up two small techniques that solve a very common performance problem.

How JavaScript manages memory

JavaScript allocates memory for every value you create, and automatically frees it later through a background process called garbage collection. The core idea it uses is reachability: a value is kept in memory as long as something can still get to it, starting from a set of always-reachable "roots" (global variables, and anything currently on the call stack).

[[code lang="js"]] let user = { name: "Ava" }; // reachable — referenced by `user` user = null; // the object is now unreachable — nothing points to it // the garbage collector will eventually free that memory [[/code]]

This is also exactly why closures (see Closures) can keep variables alive long after their outer function has returned — as long as an inner function still references them, they're still reachable, so they aren't collected.

Common ways to accidentally leak memory

A "leak" happens when something stays reachable that you actually meant to be done with. A few classic causes:

  • Forgotten timers or listeners. A setInterval or an event listener that's never cleared keeps referencing everything its callback closes over, indefinitely.
  • Growing caches with no eviction. An object or Map used as a cache that only ever grows, never removing old entries.
  • Detached DOM nodes. Removing an element from the page with .remove() but still holding a reference to it in a variable — the DOM node is gone visually, but not memory-wise.
[[code lang="js"]] function startPolling() { const id = setInterval(() => { console.log("polling..."); }, 1000); // if `id` is never passed to clearInterval, this runs forever, // keeping everything the callback references alive too } [[/code]]

Tip

Whenever you create a timer or add an event listener, ask where it gets cleaned up — clearInterval/clearTimeout, or removeEventListener (see Events & Delegation). If the answer is "nowhere," that's worth a second look.

Debouncing: waiting for a pause

Some events fire far more often than you actually need to react to — input on every keystroke, resize dozens of times a second. Debouncing delays running a function until a pause of silence has passed, canceling any earlier scheduled run.

[[code lang="js"]] function debounce(fn, delay) { let timeoutId; return function (...args) { clearTimeout(timeoutId); // cancel the previous pending call timeoutId = setTimeout(() => fn(...args), delay); }; } const search = debounce(function (query) { console.log("Searching for:", query); }, 300); // Rapid calls while typing only actually run `search` once, // 300ms after the user stops typing input.addEventListener("input", e => search(e.target.value)); [[/code]]

Throttling: at most once per interval

Throttling is similar but different: instead of waiting for silence, it guarantees a function runs at most once every fixed interval, no matter how often it's triggered.

[[code lang="js"]] function throttle(fn, interval) { let lastRun = 0; return function (...args) { const now = Date.now(); if (now - lastRun >= interval) { lastRun = now; fn(...args); } }; } const onScroll = throttle(function () { console.log("scroll position:", window.scrollY); }, 200); window.addEventListener("scroll", onScroll); // runs at most every 200ms [[/code]]

Note

Debounce = "wait until they stop" (search-as-you-type). Throttle = "run regularly, but not too often" (scroll position tracking). Picking the wrong one is a common source of janky UI.

Minimizing unnecessary DOM work

Reading and writing to the DOM (see The DOM) is far slower than plain JavaScript operations, because the browser may need to recalculate layout and repaint the screen. Batch your changes instead of interleaving many small reads and writes:

[[code lang="js"]] // Slower — forces the browser to recalculate layout on every iteration items.forEach(item => { list.innerHTML += `
  • ${item}
    • `; }); // Faster — build the full string once, touch the DOM a single time const html = items.map(item => `
  • ${item}
    • `).join(""); list.innerHTML = html; [[/code]]

    Key Takeaways

    • JavaScript frees memory automatically based on reachability — a value is kept alive only as long as something can still reference it.
    • Forgotten timers, listeners, and unbounded caches are the most common real-world memory leaks.
    • Debounce delays a function until activity pauses (search input); throttle caps it to run at most once per interval (scroll handlers).
    • Batch DOM reads/writes together instead of interleaving many small ones inside a loop.
